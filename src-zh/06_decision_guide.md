# 第六章：决策指南 — 选择合适的记忆系统

架构看过了，技术也懂了，基准测试的数据也摆在那儿。接下来最实际的问题就是：到底该选哪个？本章给你一套决策框架，按场景直接给建议。

---

## 先想清楚：你要的是哪种记忆？

| 记忆类别 | 含义 | 示例系统 |
|-------------|---------------|-----------------|
| **个性化** | 记住每位用户的偏好和过往 | Mem0, RetainDB, Honcho |
| **机构知识** | 从文档和数据中沉淀领域知识 | Cognee, Zep/Graphiti |
| **智能体学习** | 智能体从自身经验中持续进化 | Hindsight, Letta, OpenViking |
| **统一上下文** | 统一管理记忆、知识与技能 | OpenViking, ByteRover, Supermemory |

实际项目中，你多半需要组合使用。客服机器人既要认人（个性化），又要懂产品（机构知识）。编程智能体既要记住架构决策（智能体学习），又要管好项目文件和记忆（统一上下文）。

---

## 决策树

```
起点：你在做什么类型的产品？
│
├─► 面向消费者的聊天产品（用户量大，个性化需求简单）
│   │
│   ├─► 追求最快上线？→ RetainDB（3 行代码）
│   ├─► 要开源、有口碑的方案？→ Mem0（最大社区）
│   └─► 要最深度的用户理解？→ Honcho（辩证推理）
│
├─► 编程智能体 / 开发者工具
│   │
│   ├─► 本地优先、基于文件？→ ByteRover（Context Tree）
│   ├─► 要完整的智能体框架？→ Letta（记忆 + 智能体运行时）
│   └─► 跨工具共享记忆？→ OpenViking（文件系统范式）
│
├─► 企业知识系统
│   │
│   ├─► 时间线推理是刚需？→ Zep/Graphiti（双时态知识图谱）
│   ├─► 需要对接多种数据源？→ Cognee（38+ 连接器）
│   └─► 想要一站式平台？→ Supermemory（记忆 + RAG + 连接器）
│
├─► 长周期自主智能体
│   │
│   ├─► 要可解释的推理过程？→ Hindsight（4 网络 + 反思）
│   ├─► 要完整的智能体操作系统？→ Letta（记忆即操作系统范式）
│   └─► 游戏 / 仿真？→ Hindsight 或 Letta
│
└─► 轻量级 / 边缘 / 离线
    │
    └─► 只需存结构化事实？→ Nuggets/HRR（零依赖）
```

## 基于场景的推荐

### 场景 1：「我的聊天机器人需要记住用户」

**首选方案：** Mem0、RetainDB 或 Honcho

| 如果你… | 选择 | 原因 |
|-----------|--------|-----|
| 要最经得起生产考验的 | **Mem0** | 38K+ star，文档齐全，社区活跃 |
| 要最快跑起来 | **RetainDB** | 3 行代码，30 分钟跑通 |
| 要最深度的个性化 | **Honcho** | 辩证推理，自动用户建模 |
| 要免费自托管 | **Mem0** | Apache 2.0，可本地运行 |

**架构模式：**
```
用户消息 → 记忆检索 → 上下文组装 → LLM → 响应 → 记忆更新
```

### 场景 2：「构建能记住项目上下文的编程智能体」

**首选方案：** ByteRover、Letta 或 OpenViking

| 如果你… | 选择 | 原因 |
|-----------|--------|-----|
| 要基于文件、可审计的记忆 | **ByteRover** | Markdown 文件，类 git 版本控制 |
| 要一体化智能体框架 | **Letta** | 智能体 + 记忆 + 部署一站式方案 |
| 要统一上下文（记忆 + 资源 + 技能） | **OpenViking** | 文件系统范式，分层加载 |
| 要集成 Cursor/Claude Code | **ByteRover** 或 **OpenViking** | 两者均支持 MCP |

### 场景 3：「企业知识管理，而且要能做时序推理」

**首选方案：** Zep/Graphiti 或 Cognee

| 如果你… | 选择 | 原因 |
|-----------|--------|-----|
| 要支持「这什么时候变的？」这类查询 | **Zep/Graphiti** | 双时态知识图谱 |
| 要对接多种数据源 | **Cognee** | 38+ 连接器，ECL 管道 |
| 既有对话数据又有业务数据 | **Zep** | Graphiti 两手都抓 |
| 只要开源图引擎 | **Graphiti** | 独立组件，不依赖 Zep 云 |

### 场景 4：「我就是要跑分最高」

**首选方案：** Hindsight 或自己造轮子

看数据说话：
- **LongMemEval：** Agentmemory V4（96.2%）是定制方案；Hindsight（91.4%）是现成框架里最强的
- **LoCoMo：** ByteRover（92.2%）领跑；Hindsight（89.6%）紧随其后

如果跑分就是你的第一优先级：
1. 从 **Hindsight** 入手——现成框架里准确率最高，MIT 许可证
2. LoCoMo 类型的工作负载可以看 **ByteRover**
3. 要冲绝对天花板，就得自建（参考 Agentmemory V4 的 6 信号混合检索管道）

### 场景 5：「我不想东拼西凑，要一站式方案」

**首选方案：** Supermemory

Supermemory 把记忆、RAG、用户画像、提取器、连接器全装进一个 API。代价是单项能力没有专精方案深，但胜在省心。

需要自托管的话，**OpenViking** 是开源阵营中最接近一站式的选择。

## 自建 vs 用框架

### 什么时候该自己造

- 记忆需求高度领域化（医疗、法律、金融等）
- 需要对管道每个环节有完全的掌控力
- 团队里有 3 个以上工程师能持续投入维护
- 用户规模撑得起这笔投入（百万级以上）
- 在为特定基准或评估指标做极致优化

### 什么时候该用现成框架

- 要在几周内而不是几个月后上线
- 需求和现有框架的能力基本匹配
- 想搭社区持续迭代的便车
- 没有专人维护记忆基础设施
- 你的目标是做产品，不是做记忆系统

### 折中路线

很多生产系统走的是混合路线：
1. 用框架搭核心管道（比如 Mem0 或 Hindsight 负责提取 + 检索）
2. 领域相关的记忆操作用自定义逻辑补充
3. 基础设施层用托管服务（向量数据库、图数据库）

## 成本一览

| 系统 | 免费额度 | 付费起步价 | 可自托管 |
|--------|-----------|------------------|-------------|
| Mem0 | 开源 | $249/月（Pro 含图功能） | 是 |
| OpenViking | 开源 | 免费（云部署） | 是 |
| Hindsight | 开源 | 云定价待定 | 是（Docker） |
| ByteRover | CLI 免费 | SOC 2 云（定制） | 是（本地） |
| Zep/Graphiti | Graphiti 开源 | 1K 免费额度，$25/月 | 仅 Graphiti |
| Supermemory | 1M token/月 | $19/月 Pro | 仅企业版 |
| Honcho | $100 免费额度 | $2/百万 token | 是 |
| Cognee | 开源 | $35/月 开发者版 | 是 |
| RetainDB | 10K 操作/月 | 按用量计费 | 企业版 |
| Letta | 开源 | 云定价待定 | 是 |

### 隐性成本

1. **记忆操作的 LLM 调用：** 大多数系统用 GPT-4o-mini 或类似模型做提取和整理。量一大，每月 $100–1,000+ 很正常。
2. **向量数据库托管：** 不自建的话，托管费每月 $50–500 不等，看规模。
3. **图数据库托管：** Neo4j Aura 起步约 $65/月。用 Zep 必须配，Mem0g 可选。
4. **工程师时间：** 自托管就得有人看着。大概要占一名工程师 10–20% 的精力。

## 迁移路径

已经在用某个方案，想换？参考下面的迁移难度：

| 从 | 到 | 迁移难度 |
|------|----|---------------------|
| 无记忆 → Mem0 | 简单 | 快速入门文档齐全 |
| 无记忆 → RetainDB | 非常简单 | 3 行代码 |
| Mem0 → Hindsight | 中等 | 抽象方式不同，要把记忆类型映射到网络 |
| Supermemory → RetainDB | 简单 | RetainDB 有迁移指南 |
| MEMORY.md 文件 → Honcho | 简单 | Honcho 能自动检测并协助迁移 |
| MEMORY.md 文件 → OpenViking | 简单 | OpenViking 支持 markdown 导入 |
| MemGPT → Letta V1 | 中等 | 架构改了（heartbeats → 原生推理） |
| 任意 → ByteRover | 简单 | `brv curate -f ~/notes/MEMORY.md` 导入现有文件 |

---

## 总结

**刚起步：** 直接上 Mem0（社区最大，文档最全）或 RetainDB（集成最快）。

**拼准确率：** 选 Hindsight（现成框架里跑分最高）。

**做编程智能体：** ByteRover（专门为此而生）或 Letta（完整框架）。

**企业知识管理：** Zep/Graphiti（时序推理强）或 Cognee（知识图谱）。

**想一个 API 搞定一切：** Supermemory（功能面最广）。

**个性化是核心卖点：** Honcho（用户建模最深）。

**想从原理学起：** 读完本书，拿 Mem0 开源代码搭个原型，亲手跑一遍写入–管理–读取循环，一切都会豁然开朗。
